# blocker.1: Failhide in reconstructAssistantItems: catching JSON.parse errors without rethrowing

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/atoms/genBrainAtom.ts

In reconstructAssistantItems, the try/catch block catches all errors from JSON.parse and assumes they indicate non-JSON content. This hides potential real errors that aren't related to JSON parsing. The rule requires that only allowlisted errors be caught and handled carefully, while rethrowing unexpected errors. Here, any error (including unexpected ones) is silently caught and treated as plain text, violating fail-fast.

Change: Explicitly check for SyntaxError (the expected error for invalid JSON) and rethrow any other errors.

**snippet**:
```ts
  try {
    const parsed = JSON.parse(exchangeOutput);
    if (Array.isArray(parsed)) {
      // check if it looks like response output items (has function_call or message)
      const hasResponseItems = parsed.some(
        (item) =>
          item?.type === 'function_call' ||
          item?.type === 'message' ||
          item?.type === 'reasoning',
      );
      if (hasResponseItems) {
        // spread the items as-is (function_call items preserved)
        return parsed as OpenAI.Responses.ResponseInputItem[];
      }
    }
  } catch {
    // not JSON, treat as plain text
  }
```

# blocker.2: Failhide in output parsing when tools are used: catching parse errors and returning null

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/atoms/genBrainAtom.ts

In the output parsing logic for tool-enabled cases, try/catch blocks catch JSON.parse and schema.parse errors broadly, attempt extraction, and return null if parsing fails. This hides errors that indicate malformed content or unexpected failures, violating fail-fast by silently degrading instead of alerting to issues. The rule forbids catching errors without an allowlist and proper handling.

Change: Replace broad catches with specific error type checks (SyntaxError for JSON, ZodError for schema) and rethrow unexpected errors.

**snippet**:
```ts
        // when tools are plugged, we skip response_format
        // model may return JSON or plain text - try JSON first, fallback to text
        if (hasTools && isObjectSchema) {
          try {
            // try to parse as JSON first
            return askInput.schema.output.parse(JSON.parse(content));
          } catch {
            // model returned plain text - try to extract JSON from response
            const jsonMatch = content.match(/\{[\s\S]*\}/);
            if (jsonMatch) {
              try {
                return askInput.schema.output.parse(JSON.parse(jsonMatch[0]));
              } catch {
                // extracted JSON didn't match schema
              }
            }
            // if tool calls present, this is expected (partial response)
            if (calls) return null;
            // no structured output available with tools
            return null;
          }
        }
```

# blocker.3: Failhide in standard output parsing: catching parse errors when tool calls present

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/atoms/genBrainAtom.ts

In the standard output parsing block, the try/catch catches parse errors broadly, then checks error types. However, the initial catch in the tool-enabled path (above) and the nested catches don't properly rethrow unexpected errors. This violates the rule by potentially hiding real errors that aren't SyntaxError or ZodError.

Change: Ensure all catches explicitly check for allowlisted error types and rethrow unexpected ones before falling back.

**snippet**:
```ts
        // standard parsing (with response_format or string schema)
        try {
          return isObjectSchema
            ? askInput.schema.output.parse(JSON.parse(content))
            : askInput.schema.output.parse(content);
        } catch (error) {
          // allowlist: SyntaxError or ZodError - if tool calls present, expected
          const isParseError =
            error instanceof SyntaxError ||
            error?.constructor?.name === 'ZodError';
          if (!isParseError) throw error;
          if (calls) return null;
          throw new Error('structured output parse failed with no tool calls');
        }
```
