# research: external product claims

## tool definition and invocation

### [FACT] strict mode ensures schema adherence

> "Setting strict to true will ensure function calls reliably adhere to the function schema"
> "It is recommended to always enable strict mode" [1]

strict mode leverages structured outputs and requires `additionalProperties: false` for each object in parameters.

### [FACT] tool_choice controls function invocation

> "The replacement for functions is the tools parameter, and the replacement for function_call is the tool_choice parameter" [1]

tool_choice accepts:
- `"auto"` (default) — model decides whether to call function or generate text
- `"none"` — model never calls functions, only generates text
- `"required"` — model must call at least one function

### [FACT] parallel_tool_calls enables multiple calls per turn

> "The model may choose to call multiple functions in a single turn. You can prevent this by setting parallel_tool_calls to false, which ensures exactly zero or one tool is called" [1]

### [FACT] tool limits are well-defined

> "As of May 2025, any setup with fewer than ~100 tools and fewer than ~20 arguments per tool is considered in-distribution and should perform within expected reliability bounds" [2]

### [FACT] structured outputs vs tools serve different purposes

> "Function calling is useful when you are building an application that bridges the models and functionality of your application"
> "Structured Outputs via response_format are more suitable when you want to indicate a structured schema for use when the model responds to the user, rather than when the model calls a tool" [3]

## error handling and safety

### [FACT] errors should be surfaced to model

> "When tool execution fails, instead of providing the tool output to the model, provide the error to the model, though for most models you need to provide some guidelines in the system prompt" [1]

### [SUMP] tools should be idempotent for retry safety

> "Database tools need the Idempotent Operation pattern because agents retry on timeout and tools must handle duplicate calls gracefully" [4]

> "Side-effectful tools should accept idempotency tokens and be safe under retries; read paths should be pure" [4]

### [SUMP] error responses should guide agent recovery

> "A response that says 'rate limited, retry after 30 seconds or reduce batch size to 50' gives the agent a path forward. This is far more useful than raw error codes" [5]

### [SUMP] action layer should govern tool execution

> "An agent action layer governs tool calls by enforcing schemas, auth, retries, idempotency, and observability, ensuring that probabilistic LLM tool calls remain safe in production" [6]

## agentic patterns

### [FACT] responses can contain multiple tool calls

> "Since model responses can include zero, one, or multiple calls, it is best practice to assume there are several" [1]

### [SUMP] tool order may require explicit guidance

> "o3/o4-mini are trained to accomplish goals with tools, but can make mistakes in the order of tool calls, so it is recommended to explicitly outline the orders to accomplish certain tasks" [2]

### [SUMP] tool filtering reduces payload size

> "Developers should filter tools to avoid ballooning payloads by taking advantage of the allowed_tools parameter to use only necessary tools and save on unnecessary context" [2]

### [OPIN] function calling is structured output

> "Function calling is a specific type of structured output where the LLM tells your system which function to run and provides the parameters in a validated format" [7]

## safety and production patterns

### [SUMP] five patterns make agents safe in production

> "circuit breakers for LLM quality failures, validation gates before tool execution, idempotent workflows with saga rollbacks, token and cycle budget guardrails, and human escalation for high-risk actions" [6]

### [SUMP] typed messages tame open-ended model behavior

> "Primary means to tame open-ended model behaviour are typed and schema-validated messages; explicit capability scopes for tools; idempotent and (where feasible) transactional semantics; rate/authority limits. All of these convert free-form model outputs into predictable, auditable actions" [8]

### [SUMP] tool composition requires consistent response shapes

> "Consistent response shapes so one tool's output feeds cleanly as another's input, batch support so agents don't loop one-at-a-time, and multiple abstraction levels so the agent picks the right granularity for the task" [5]

## open questions

### [KHUE] parallel tool calls may not always work

> "There's a community report that some users experimenting with the Responses API only see one function call per response, even though they've set parallel_tool_calls: true" [9]

should we test for and document parallel invocation behavior?

### [KHUE] how should tool execution metrics be captured?

rhachet's `BrainPlugToolExecution` requires `metrics: { cost: { time: IsoDuration } }`. callers must track execution time — is this burden acceptable? should the tool declaration factory auto-wrap with timing?

### [KHUE] how should tool execution errors be classified?

rhachet distinguishes `error:constraint` (bad input) from `error:malfunction` (system failure). how does this map to practical error scenarios? should we document canonical error patterns?
