# research: external product references

## rhachet tool contracts

### BrainPlugToolDefinition [1]

> "a declaration of a tool that can be invoked by the brain"
> "enables callers to plug tools into BrainAtom and BrainRepl for tool use"

key properties:
- `slug` — unique identifier (e.g., `"stripe.customer.lookup"`)
- `name` — human-readable display name
- `description` — helps brain understand when to use tool
- `schema.input` — zod schema for input validation
- `schema.output` — zod schema for output validation

grain determines execute presence:
- `'atom'` — no execute (caller handles execution)
- `'repl'` — execute required (repl calls it automatically)

### BrainPlugToolInvocation [2]

> "a request from the brain to execute a tool"
> "maps to anthropic's tool_use block and openai's tool_calls entry"

key properties:
- `exid` — provider-assigned external ID for correlation
- `slug` — which tool to call
- `input` — parsed arguments, typed via TInput

### BrainPlugToolExecution [3]

> "the result of a tool execution, ready to feed back into the brain"
> "maps to anthropic's tool_result block and openai's tool message"

discriminated union by `signal`:
- `signal: 'success'` — output: TOutput
- `signal: 'error:constraint'` — output: { error: Error } (bad input)
- `signal: 'error:malfunction'` — output: { error: Error } (system failure)

required metrics:
- `metrics.cost.time` — IsoDuration for execution time

## rhachet operations

### genBrainPlugToolDeclaration [4]

> "factory to create a BrainPlugToolDefinition with type inference from schema"
> "wraps execute to return BrainPlugToolExecution with time and signal"
> "classifies errors: BadRequestError → 'error:constraint', else → 'error:malfunction'"

usage:
```ts
const customerLookup = genBrainPlugToolDeclaration({
  slug: 'customer-lookup',
  name: 'Customer Lookup',
  description: 'lookup customer by email',
  schema: {
    input: z.object({ email: z.string().email() }),
    output: z.object({ id: z.string(), name: z.string() }),
  },
  execute: async ({ invocation }) => {
    // invocation.input typed as { email: string }
    // return type enforced as { id: string, name: string }
    return { id: 'cus_123', name: 'Alice' };
  },
});
```

### asBrainPlugToolDict [5]

> "converts an array of tools into a typed dictionary keyed by slug"
> "enables O(1) tool lookup by slug with per-slug type preservation"

usage:
```ts
const tools = [searchTool, calcTool] as const;
const toolDict = asBrainPlugToolDict(tools);

// lookup by known slug — gets specific type
const search = toolDict['search'];  // typeof searchTool

// lookup by dynamic slug — gets union type
const tool = toolDict[invocation.slug];  // union | undefined
```

## rhachet brain grains

### brain.atom vs brain.repl [6]

> "○ atoms are stateless. one turn in, one turn out."
> "↻ repls loop until complete."

| grain | symbol | tool execution |
|-------|--------|----------------|
| BrainAtom | ○ | outputs invocations (caller executes) |
| BrainRepl | ↻ | executes tools internally |

## openai tool contract

### strict mode requirements [7]

> "strict mode works by leveraging the structured outputs feature"
> "requires additionalProperties to be set to false for each object in the parameters"

to enable strict mode:
1. add `"strict": true` attribute
2. add `"additionalProperties": false` for each object
3. mark all fields in properties as required

### function definition schema [8]

```json
{
  "type": "function",
  "name": "get_weather",
  "description": "get current temperature for a location",
  "parameters": {
    "type": "object",
    "properties": {
      "location": { "type": "string" }
    },
    "required": ["location"],
    "additionalProperties": false
  },
  "strict": true
}
```

### function_call output item [8]

```json
{
  "type": "function_call",
  "call_id": "call_abc123",
  "name": "get_weather",
  "arguments": { "location": "Paris" }
}
```

### function_call_output input item [8]

```json
{
  "type": "function_call_output",
  "call_id": "call_abc123",
  "output": "72°F, sunny"
}
```

> "Tool calls and outputs use call_id correlation rather than sequential message pairing"
