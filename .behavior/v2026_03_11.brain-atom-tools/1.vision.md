# vision: brain atom tool use

## the outcome world

### day-in-the-life

a developer who builds an agentic workflow needs their brain atom to call external tools — lookup a customer in stripe, fetch weather data, query a database.

**before**: they either:
- use a full repl (heavyweight, spins up sandbox, overkill for simple tool use)
- manually orchestrate a loop: call atom → parse response → detect tool request → execute → call atom again

**after**: they plug tools into the atom, call `.ask()`, and receive either:
- direct output (brain answered without tools)
- tool invocations (brain wants to call tools)

```ts
import { genBrainPlugToolDeclaration, asBrainPlugToolDict } from 'rhachet';

// define tool with typed schemas
const weatherLookup = genBrainPlugToolDeclaration({
  slug: 'weather.lookup',
  name: 'Weather Lookup',
  description: 'get current weather for a city',
  schema: {
    input: z.object({ city: z.string() }),
    output: z.object({ temp: z.number(), conditions: z.string() }),
  },
  execute: async ({ invocation }) => {
    const weather = await fetchWeather(invocation.input.city);
    return { temp: weather.temp, conditions: weather.conditions };
  },
});

// build tool dict for O(1) lookup
const tools = [weatherLookup];
const toolDict = asBrainPlugToolDict(tools);

// initial prompt
let result = await brainAtom.ask({
  prompt: 'what is the current weather in austin?',
  schema: { output: z.object({ summary: z.string() }) },
  plugs: { tools },
});

// loop until output is defined
while (result.calls) {
  // execute each invocation
  const executions = await Promise.all(
    result.calls.tools.map(async (invocation) => {
      const tool = toolDict[invocation.slug];
      return tool.execute({ invocation }, context);
    }),
  );

  // continue episode with tool results
  result = await brainAtom.ask({
    on: { episode: result.episode },
    prompt: executions,  // BrainPlugToolExecution[]
    schema: { output: z.object({ summary: z.string() }) },
    plugs: { tools },
  });
}

// result.output is now defined
console.log(result.output.summary);
```

### before/after contrast

| before | after |
|--------|-------|
| atoms are inference-only | atoms support tool use natively |
| tool use requires repl overhead | tool use is lightweight via atoms |
| output always present | output nullable when brain defers |
| calls always null | calls contains tool invocations when plugged |

### the "aha" moment

> "wait — i can just plug tools into the atom and it tells me which ones to call? i don't need to parse json or guess at the format?"

the value clicks when the developer realizes:
1. tool definitions use zod schemas (same as output)
2. tool invocations come back parsed and typed
3. the type system narrows: no tools → no null checks needed

## user experience

### usecases

| goal | approach |
|------|----------|
| simple inference (no tools) | `ask()` without plugs → output always defined |
| single-turn tool use | plug tools → check `result.calls` → execute → continue episode |
| multi-turn agentic loop | plug tools → loop until `result.output` is defined |
| hybrid: answer or tool | plug tools → brain decides whether to answer directly or request tools |

### grain-based behavior

| grain | symbol | tool execution | `calls` output |
|-------|--------|----------------|----------------|
| BrainAtom | ○ | outputs invocations (caller executes) | `{ tools: [...] }` or `null` |
| BrainRepl | ↻ | executes tools internally via `execute()` | always `null` |

**this implementation focuses on BrainAtom** — the atom outputs tool invocations and the caller is responsible for execution and continuation.

### contract inputs

```ts
// tool definition (input) — note: has BOTH input AND output schemas
interface BrainPlugToolDefinition<TInput, TOutput, TGrain, TSlug> {
  slug: TSlug;                         // e.g., "stripe.customer.lookup"
  name: string;                        // display name for UI/logs
  description: string;                 // aids brain in tool selection
  schema: {
    input: z.Schema<TInput>;           // zod schema for input validation
    output: z.Schema<TOutput>;         // zod schema for output validation
  };
  // execute only present for 'repl' grain (repl runs execution internally)
  execute?: (input: { invocation }, context) => Promise<BrainPlugToolExecution>;
}

// plugs (input)
interface BrainPlugs {
  tools?: BrainPlugToolDefinition[];
  memory?: never;   // future
  access?: never;   // future
}

// tool execution result (input for continuation)
type BrainPlugToolExecution<TInput, TOutput> =
  | { exid: string; slug: string; input: TInput; signal: 'success'; output: TOutput; metrics: { cost: { time: IsoDuration } } }
  | { exid: string; slug: string; input: TInput; signal: 'error:constraint' | 'error:malfunction'; output: { error: Error }; metrics: { cost: { time: IsoDuration } } };

// prompt type (conditional based on plugs)
type AsBrainPromptFor<TPlugs> = TPlugs extends { tools: [...] }
  ? string | BrainPlugToolExecution[]  // string OR tool results
  : string;                            // string only
```

### rhachet operations

```ts
import { genBrainPlugToolDeclaration, asBrainPlugToolDict } from 'rhachet';

// create tool with automatic type inference and error classification
const customerLookup = genBrainPlugToolDeclaration({
  slug: 'customer-lookup',
  name: 'Customer Lookup',
  description: 'lookup customer by email',
  schema: {
    input: z.object({ email: z.string().email() }),
    output: z.object({ id: z.string(), name: z.string() }),
  },
  execute: async ({ invocation }, context) => {
    // invocation.input typed as { email: string }
    // BadRequestError → 'error:constraint', else → 'error:malfunction'
    const customer = await context.stripe.customers.list({ email: invocation.input.email });
    if (!customer.data[0]) throw new BadRequestError('customer not found');
    return { id: customer.data[0].id, name: customer.data[0].name };
  },
});

// O(1) tool lookup by slug with type preservation
const toolDict = asBrainPlugToolDict([customerLookup, invoiceTool]);
const tool = toolDict['customer-lookup'];  // preserves specific type
```

### contract outputs

```ts
// tool invocation (output)
interface BrainPlugToolInvocation<TInput> {
  exid: string;    // provider-assigned id for correlation
  slug: string;    // which tool to call
  input: TInput;   // parsed arguments
}

// brain output (output)
interface BrainOutput<TOutput, TPlugs> {
  output: TOutput | null;           // null when brain defers to tools
  calls: { tools: [...] } | null;   // tool invocations when plugged
  metrics: BrainOutputMetrics;
  episode: BrainEpisode;            // for continuation
  series: null;                     // atoms don't support series
}
```

### timeline

1. **define tools** — create tool definitions with zod schemas
2. **call ask()** — pass plugs.tools to atom
3. **check result** — either `output` is defined (done) or `calls` has invocations
4. **execute tools** — if calls present, execute and continue episode
5. **loop or return** — repeat until output is defined

## mental model

### how users would describe this to a friend

> "you give the brain a list of tools it can use. when you ask it a question, it either answers directly or tells you which tool to call. you execute the tool, feed the result back, and continue until it answers."

### analogies

| analogy | map |
|---------|---------|
| restaurant waiter | brain = waiter; tools = kitchen stations; invocations = orders |
| function dispatch | brain = dispatcher; tools = registered handlers; invocations = calls |
| unix pipes | brain = shell; tools = programs; invocations = command invocations |

### terms comparison

| user term | rhachet term |
|-----------|--------------|
| "function" | tool (BrainPlugToolDefinition) |
| "function call" | tool invocation (BrainPlugToolInvocation) |
| "arguments" | input |
| "return value" | tool execution (BrainPlugToolExecution) |

## evaluation

### how well does it solve the goals?

| goal | status |
|------|--------|
| enable tool use in atoms | ✓ direct support via plugs.tools |
| type safety | ✓ zod schemas for both definition and invocation |
| provider agnostic | ✓ rhachet contract abstracts openai specifics |
| progressive complexity | ✓ no tools = no null checks |

### pros

- **lightweight** — no repl overhead for simple tool use
- **type safe** — zod schemas throughout
- **composable** — tool definitions are reusable
- **explicit** — brain's intent is clear in result.calls
- **continuation** — episode enables multi-turn tool loops

### cons

- **manual loop** — caller must implement tool execution loop (vs repl which handles it)
- **no sandbox** — tools execute in caller's process (vs repl's isolated sandbox)
- **metrics required** — caller must track execution time for `BrainPlugToolExecution.metrics`

### edgecases and pit of success

| edgecase | pit of success |
|----------|----------------|
| no tools plugged | `calls` is typed as `null`, no null checks needed |
| tools plugged but unused | `output` defined, `calls` null — works naturally |
| tools plugged and used | `output` null, `calls` defined — type system enforces check |
| multiple tool calls | `calls.tools` is array — all invocations available |
| invalid tool input | zod schema validation catches at parse time |
| tool execution error | `BrainPlugToolExecution.signal` distinguishes success from error |
| tool result continuation | `prompt` accepts `BrainPlugToolExecution[]` when tools plugged |

## open questions & assumptions

### validated via research

all open questions have been answered via research on 2026-03-11:

| question | answer |
|----------|--------|
| tool definition format | `{ type: "function", name, description, parameters, strict: true }` — zod → JSON Schema works |
| tool calls in response | `{ type: "function_call", id, name, arguments }` — maps to `BrainPlugToolInvocation { exid, slug, input }` |
| tool results submission | requires explicit `function_call_output` items with `call_id` — prompt injection won't work |
| output schema + tools | **mutually exclusive** — `response_format` OR `tools`, not both |

### key discovery: output schema + tools are mutually exclusive

openai docs recommend one or the other:
- `response_format` = for structured final output to user
- `tools` = for function calls to external systems

**implication**: when tools are plugged, we do NOT use `response_format`. the wisher's "w/ and w/o output schema" means:
- test tool use WITHOUT output schema (tools mode)
- test inference WITH output schema (extant behavior, no tools)

### key discovery: tool results need explicit support — implemented ✓

the responses api requires `function_call_output` items with `call_id` for tool result continuation. prompt injection (text in prompt) is not the correct format.

**rhachet implemented** `BrainPlugToolExecution` type and `AsBrainPromptFor<TPlugs>` conditional type. when tools are plugged, the `prompt` parameter accepts `BrainPlugToolExecution[]` for tool result continuation. this maps directly to openai's `function_call_output` format.

## what is awkward?

### tool result injection — resolved ✓

**research confirmed**: tool results via prompt string is incorrect for openai responses api. openai requires explicit `function_call_output` items.

**rhachet solved this** with `BrainPlugToolExecution` and `AsBrainPromptFor<TPlugs>`:

```ts
// when tools are plugged, prompt accepts BrainPlugToolExecution[]
const result2 = await atom.ask({
  on: { episode: result1.episode },
  prompt: [{
    exid: invocation.exid,
    slug: invocation.slug,
    input: invocation.input,
    signal: 'success',
    output: customer,
    metrics: { cost: { time: 'PT0.5S' } },
  }],
  schema: { output: z.object({ customer: CustomerSchema }) },
  plugs: { tools },
});
```

this elegantly reuses the `prompt` parameter:
- `string` for initial prompt or follow-up questions
- `BrainPlugToolExecution[]` for tool result continuation

the contract also supports error signals (`error:constraint`, `error:malfunction`) so the brain can reason about tool failures.

### no automatic loop

caller must implement the tool loop manually. the repl handles this automatically.

**tradeoff**: this is intentional — atoms are single-turn by design. the loop is the caller's responsibility. this keeps atoms simple and composable.

### schema for tool result continuation

when we continue with tool results, what schema do we use? the original output schema? or a special "awaits tool result" schema?

**assumption**: use the same output schema. if brain still needs tools, it will request them again. if done, it will produce output.
