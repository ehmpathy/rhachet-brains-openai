## patterns identified

### [REUSE] config-based model selection

> "brain spec configuration by atom slug"
> "maps slugs to api model names, descriptions, and specs" [1]

```ts
export const CONFIG_BY_ATOM_SLUG: Record<OpenaiBrainAtomSlug, BrainAtomConfig> = {
  'openai/gpt/4o': {
    model: 'gpt-4o',
    description: 'gpt-4o - multimodal model for reason and vision',
    spec: { ... },
  },
  // ...
};
```

**reuse**: the config pattern already supports `skills: { tooluse: true }` in spec.gain. no change needed.

### [REUSE] dependency injection via context

> "get openai client from context or create new one" [2]

```ts
const openai =
  (context?.openai as OpenAI | undefined) ??
  new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
```

**reuse**: same pattern works for tool execution. tools can receive context for their dependencies.

### [REUSE] zod to JSON Schema conversion

> "convert a zod schema to JSON schema for native SDK enforcement" [3]

```ts
export const asJsonSchema = (input: { schema: z.ZodSchema }): object => {
  return z.toJSONSchema(input.schema, { target: 'openAi' });
};
```

**reuse**: this pattern directly applies to tool parameter schemas. already uses `{ target: 'openAi' }` for openai compatibility.

---

### [REUSE] episode continuation via prior exchanges

> "build input from prior episode exchanges + current prompt" [2]

```ts
const priorMessages: OpenAI.Responses.ResponseInputItem[] =
  askInput.on?.episode?.exchanges.flatMap((exchange) => [
    { role: 'user' as const, content: exchange.input },
    { role: 'assistant' as const, content: exchange.output },
  ]) ?? [];
```

**reuse**: the episode pattern continues to work. tool invocations and results become additional exchange items.

### [EXTEND] BrainAtomPlugs type

> "plugs?: BrainAtomPlugs" [2]

the `BrainAtomPlugs` type is referenced but not used for tools. this needs extension to handle `plugs.tools`.

**extend**: add tool support when `plugs.tools` is provided:
1. convert tool definitions to openai function format
2. pass to responses api via `tools` parameter
3. parse `function_call` items from response output
4. return `calls.tools` when tool invocations present

---

### [EXTEND] output extraction from response

> "extract content from responses api output" [2]

```ts
const outputItem = response.output.find(
  (item) => item.type === 'message',
);
```

current code only handles `type === 'message'` items.

**extend**: also handle `type === 'function_call'` items:
1. detect function_call items in response.output
2. parse each to BrainPlugToolInvocation
3. return null output + calls.tools when invocations present

---

### [EXTEND] prompt input type

> "prompt: string" [2]

current signature only accepts string prompts.

**extend**: accept `BrainPlugToolExecution[]` when tools are plugged:
1. conditional type: `AsBrainPromptFor<TPlugs>`
2. when array: map to `function_call_output` items
3. append to prior exchanges for continuation

---

### [REPLACE] response_format vs tools mutual exclusivity

> "text: { format: { type: 'json_schema', ... } }" [2]

current code always uses response_format for structured output.

**replace**: when tools are plugged:
1. do NOT use response_format
2. use tools parameter instead
3. structured output via tool result continuation

this matches openai guidance that response_format and tools serve different purposes and should not be combined.
