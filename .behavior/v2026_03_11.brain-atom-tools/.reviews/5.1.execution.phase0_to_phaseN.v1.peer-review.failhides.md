# blocker.1: Failhide in reconstructAssistantItems: catching JSON.parse errors without rethrowing

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/atoms/genBrainAtom.ts

In reconstructAssistantItems, the try/catch block catches JSON.parse errors and falls back to treating the output as plain text. This hides potential errors where the exchangeOutput should be valid JSON but is malformed, violating the fail-fast principle. The rule requires that errors be rethrown unless specific allowlisted errors are handled carefully. Here, all parse errors are caught and ignored, leading to silent degradation instead of failing fast to alert of invalid state.

Change: Either throw a specific error (e.g., BadRequestError) when JSON.parse fails, or refactor to avoid the try/catch if the fallback is not appropriate.

**snippet**:
```ts
const reconstructAssistantItems = (
  exchangeOutput: string,
): OpenAI.Responses.ResponseInputItem[] => {
  // try to parse as JSON array of response items
  try {
    const parsed = JSON.parse(exchangeOutput);
    if (Array.isArray(parsed)) {
      // check if it looks like response output items (has function_call or message)
      const hasResponseItems = parsed.some(
        (item) =>
          item?.type === 'function_call' ||
          item?.type === 'message' ||
          item?.type === 'reasoning',
      );
      if (hasResponseItems) {
        // spread the items as-is (function_call items preserved)
        return parsed as OpenAI.Responses.ResponseInputItem[];
      }
    }
  } catch {
    // not JSON, treat as plain text
  }
  // fallback: treat as plain assistant text
  return [{ role: 'assistant' as const, content: exchangeOutput }];
};
```

# blocker.2: Failhide in output parsing when tools are used: catching parse errors and returning null

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/atoms/genBrainAtom.ts

In the ask function's output parsing logic for tool-enabled cases, try/catch blocks catch JSON.parse and schema.parse errors, attempt extraction, and ultimately return null if parsing fails, especially when tool calls are present. This hides errors that indicate the model's output is invalid or malformed, violating fail-fast by silently returning null instead of throwing. The rule forbids catching and hiding errors; if parsing fails unexpectedly, it should throw to alert of the issue.

Change: Replace the try/catch with direct parsing and let errors propagate, or throw specific errors like UnexpectedCodePathError if the state is invalid.

**snippet**:
```ts
        // when tools are plugged, we skip response_format
        // model may return JSON or plain text - try JSON first, fallback to text
        if (hasTools && isObjectSchema) {
          try {
            // try to parse as JSON first
            return askInput.schema.output.parse(JSON.parse(content));
          } catch {
            // model returned plain text - try to extract JSON from response
            const jsonMatch = content.match(/\{[\s\S]*\}/);
            if (jsonMatch) {
              try {
                return askInput.schema.output.parse(JSON.parse(jsonMatch[0]));
              } catch {
                // extracted JSON didn't match schema
              }
            }
            // if tool calls present, this is expected (partial response)
            if (calls) return null;
            // no structured output available with tools
            return null;
          }
        }
```

This violates fail-fast because it silently ignores parsing failures that could indicate bugs or invalid responses. The rule requires rethrowing errors unless specific allowlisted errors are handled.

Change: Allow the parse to throw naturally, or add guard clauses with early throws if the conditions for partial responses aren't met.
