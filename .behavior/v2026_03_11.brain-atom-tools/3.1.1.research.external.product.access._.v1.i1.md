# research: external product access

## remote repositories

### 1. OpenAI Responses API

**what**: LLM inference endpoint with function calling support

**interface**: REST API via `openai` npm package

**contract**:

#### tool definition schema [1]

```json
{
  "type": "function",
  "name": "get_weather",
  "description": "Determine weather in my location",
  "parameters": {
    "type": "object",
    "properties": {
      "location": { "type": "string" }
    },
    "additionalProperties": false,
    "required": ["location"]
  }
}
```

> "Functions are strict by default (no `"strict": true` flag needed)" [1]

> "Structure is flattened: `name` and `description` are top-level, not nested under a `"function"` object" [1]

#### function call output item [1]

```json
{
  "type": "function_call",
  "call_id": "call_abc123",
  "name": "get_weather",
  "arguments": { "location": "Paris" }
}
```

> "Tool invocations appear as distinct Item types in the response `output` array" [1]

#### function call output submission [1]

```json
{
  "type": "function_call_output",
  "call_id": "call_abc123",
  "output": "72°F, sunny"
}
```

> "Submit results as separate Items when continuing context" [1]

> "Tool calls and outputs use `call_id` correlation rather than sequential message pairing" [1]

## best practices

### industry wide

1. **strict mode by default** [2]
   > "Setting strict to true will ensure function calls reliably adhere to the function schema"
   > "It is recommended to always enable strict mode"

2. **parallel tool calls** [2]
   > "You can prevent multiple function calls by setting parallel_tool_calls to false"

3. **call_id correlation** [1]
   > "Tool calls and outputs use `call_id` correlation rather than sequential message pairing in Chat Completions"

### within this repo

1. **dependency injection via context**
   - OpenAI client passed via `context?.openai` or created from env
   - enables testing with mock clients

2. **config-based model selection**
   - `CONFIG_BY_ATOM_SLUG` maps slugs to model configs
   - separates model concerns from operation logic

3. **zod schema conversion**
   - `z.toJSONSchema()` converts zod schemas to JSON Schema
   - same pattern can be used for tool parameter schemas

4. **responses api item structure**
   - prior exchanges mapped to `ResponseInputItem[]`
   - response parsed from `output` array items
