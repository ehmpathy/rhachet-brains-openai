## mechanisms in blueprint

### 1. tool definition conversion

**blueprint proposes**: inline map to convert plugs.tools to openai function format

```ts
const openaiFunctionTools = askInput.plugs.tools.map((tool) => ({
  type: 'function',
  name: tool.slug,
  description: tool.description,
  parameters: z.toJSONSchema(tool.schema.input),
  strict: true,
}));
```

**does codebase have this?**: no. this is new functionality.

**duplicate risk?**: none. no extant tool conversion exists.

**why it holds**:

i searched for `function_call`, `tools`, `type.*function`, and `toJSONSchema` patterns. the codebase has:
- no tool registration mechanism
- no openai function format construction
- no plugs.tools iteration

this is genuinely new functionality. the blueprint must create this conversion because:
1. rhachet's BrainPlugToolDefinition has different shape than openai's function tool
2. the conversion is local to genBrainAtom (no other file needs it)
3. extracting to a helper would add indirection without reuse opportunity

**verdict**: new mechanism required, no duplicate.

### 2. prompt type branch

**blueprint proposes**: check if prompt is string or BrainPlugToolExecution[]

```ts
const isToolExecutionArray = Array.isArray(askInput.prompt);
```

**does codebase have this?**: no. prompts are currently string-only.

**duplicate risk?**: none. this is new type discrimination.

**why it holds**:

i searched for `Array.isArray`, type guards, and prompt discrimination. the codebase:
- treats all prompts as strings
- has no conditional prompt type logic
- has no BrainPlugToolExecution reference

the blueprint introduces this because:
1. rhachet's `AsBrainPromptFor<TPlugs>` is a conditional type: `string | BrainPlugToolExecution[]`
2. we must discriminate at runtime to build correct openai input
3. `Array.isArray` is the idiomatic type guard for this discrimination

alternative considered: checking for `typeof prompt === 'string'`. chose `Array.isArray` because:
- more explicit about what the array case is
- aligns with the type: "is this the array variant?"

**verdict**: new mechanism required, no duplicate.

### 3. function_call item extraction

**blueprint proposes**: filter response.output for function_call items

```ts
const functionCallItems = response.output.filter(
  (item) => item.type === 'function_call',
);
```

**does codebase have this?**: similar pattern at genBrainAtom.ts:112-120

extant pattern:
```ts
const outputItem = response.output.find((item) => item.type === 'message');
const textContent = outputItem?.type === 'message'
  ? outputItem.content.find((c) => c.type === 'output_text')
  : undefined;
```

**analysis**:
- extant uses `find` (single item) for message
- blueprint uses `filter` (multiple items) for function_calls
- both follow the same pattern: extract by type from response.output
- the difference (find vs filter) is appropriate: message is one, function_calls can be many

**duplicate risk?**: none. same pattern, different item type.

**verdict**: consistent with extant mechanism.

### 6. JSON.stringify for output

**blueprint proposes**: `JSON.stringify(exec.output)` for function_call_output

**does codebase have this?**: no direct usage, but this is the inverse of extant JSON.parse

**why it holds**:

openai's function_call_output requires string format. the blueprint serializes tool output to string before send. this mirrors how we deserialize arguments on receipt.

symmetry:
- receive: `JSON.parse(item.arguments)` — openai sends JSON string → we parse
- send: `JSON.stringify(exec.output)` — we have object → openai wants JSON string

the pattern is correct because openai's api uses JSON strings for both directions.

**verdict**: consistent inverse of extant mechanism.
