## backwards compatibility concerns identified

### 1. calls is null when no tools plugged

**location:** `genBrainAtom.ts` line 196-203, `genBrainRepl.ts` line 240-242

**what it does:** when no tools are plugged, `calls` field is always `null`. this ensures callers who don't use tools see no difference from prior behavior.

**was this explicitly requested?**

yes. from `2.1.criteria.blackbox.md`:

```
# usecase.3 = no tools plugged

given(a brain atom without tools plugged)
  when(ask is called)
    then(output is always defined)
      sothat(extant behavior preserved)
    then(calls is always null)
      sothat(type system enforces no null checks)
```

**why it holds:** the criteria explicitly states "extant behavior preserved" — callers who don't use tools should see no change in the return type behavior. the `calls: null` value satisfies this.

**verdict:** explicitly requested, not an assumption.

### 2. plugs parameter is optional

**location:** `genBrainAtom.ts` signature

```ts
plugs?: TPlugs;
```

**what it does:** callers who don't use tools don't need to pass `plugs` parameter.

**was this explicitly requested?**

yes. from `3.3.1.blueprint.product.v1.i1.md` ask() signature change shows `plugs?: TPlugs` as optional.

**verdict:** explicitly requested, not an assumption.

### 3. graceful fallback when output parse fails with tool calls

**location:** `genBrainAtom.ts` line 260-298 output extraction

```ts
try {
  return askInput.schema.output.parse(JSON.parse(content));
} catch {
  // parse failed — if tool calls present, this is expected (partial response)
  if (calls) return null;
  throw new Error('structured output parse failed with no tool calls');
}
```

**what it does:** when brain returns text that doesn't parse AND tool calls are present, returns `null` instead of a throw.

**was this explicitly requested?**

yes. from `3.3.1.blueprint.product.v1.i1.md`:

```
### 3. independent extraction pattern
// try to parse output from text, with graceful fallback
```

**why it holds:** this is NOT backwards compat — it's new behavior for tool use scenarios. without tools, parse failure still throws as before. the fallback only activates when `calls` is present, which is a new code path.

**verdict:** explicitly requested, not backwards compat.

### 4. reconstructAssistantItems() fallback to plain text

**location:** `genBrainAtom.ts` line 33-57

```ts
const reconstructAssistantItems = (exchangeOutput: string) => {
  try {
    const parsed = JSON.parse(exchangeOutput);
    if (Array.isArray(parsed)) {
      const hasResponseItems = parsed.some(
        (item) => item?.type === 'function_call' || ...
      );
      if (hasResponseItems) return parsed;
    }
  } catch {
    // not JSON, treat as plain text
  }
  return [{ role: 'assistant', content: exchangeOutput }];
};
```

**what it does:** when exchange output is not a JSON array with function_call items, treats it as plain text.

**was this explicitly requested?**

implicitly yes. from `3.3.1.blueprint.product.v1.i1.md`:

```
├─ [○] prior messages from episode
```

**why it holds:** the `[○]` symbol means "retain" — prior message construction must continue to work. episodes created before tool use was added store exchange output as plain text strings. this function detects the format and handles both:

1. **plain text** (prior episodes, no tools) → returns `[{ role: 'assistant', content }]`
2. **JSON array** (tool use episodes) → returns the parsed items as-is

without this fallback, continuation of prior episodes would fail. the fallback is NOT "to be safe" — it is required for functional continuity.

**verdict:** required for prior episode compatibility (implicitly requested by "retain prior messages").

**verdict:** explicitly requested, not an assumption.

## deep examination: could any item be pruned?

### question: is `reconstructAssistantItems()` fallback necessary?

**test:** what happens if we remove the fallback and always try JSON parse?

```ts
// hypothetical: no fallback
const reconstructAssistantItems = (exchangeOutput: string) => {
  return JSON.parse(exchangeOutput); // throws on prior episodes
};
```

**result:** continuation of prior episodes (where `exchangeOutput` is plain text like `{"content":"hello"}`) would throw `SyntaxError: Unexpected token`. the fallback is required.

### question: is `calls: null` necessary when no tools plugged?

**test:** what if `calls` were `undefined` instead?

**result:** the rhachet `BrainOutput` interface requires `calls` to be `{ tools: [...] } | null`. `undefined` would fail type check. `null` is the correct value.
