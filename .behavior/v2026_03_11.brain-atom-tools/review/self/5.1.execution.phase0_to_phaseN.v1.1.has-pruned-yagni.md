## utility functions (emergent requirements)

these functions were not explicitly named in blueprint but are minimum viable for implied requirements:

### `asOpenaiFunctionName()` — 1 line

```ts
export const asOpenaiFunctionName = (slug: string): string =>
  slug.replace(/\./g, '_');
```

**why needed:** openai requires function names match `^[a-zA-Z0-9_-]+$`. rhachet slugs use dots (e.g., `weather.lookup`). minimum viable: convert dots to underscores.

**YAGNI check:** could not be simpler. no regex cache, no options, no validation.

### `genFunctionCallOutputId()` — 1 line

```ts
const genFunctionCallOutputId = (): string =>
  `fc_${Date.now().toString(36)}_${Math.random().toString(36).slice(2, 10)}`;
```

**why needed:** openai requires `function_call_output.id` to start with `fc_` prefix. discovered via 400 error during implementation.

**YAGNI check:** minimum viable id generation. no uuid library, no collision tracking.

### `reconstructAssistantItems()` — 20 lines

**why needed:** blueprint says "prior messages from episode" but function_call exchanges are serialized as JSON arrays. to continue with tool results, prior function_call items must be reconstructed (not treated as plain text).

**YAGNI check:** minimum viable detection + reconstruction. tries JSON parse, checks for response item types, falls back to plain text. no caching, no validation.

### `systemInstruction` — 10 lines

**why needed:** blueprint says "skip response_format (mutually exclusive with tools)". but we still need structured JSON output. solution: system instruction requesting JSON output matching schema.

**YAGNI check:** minimum viable. plain template string, no custom formatting.
