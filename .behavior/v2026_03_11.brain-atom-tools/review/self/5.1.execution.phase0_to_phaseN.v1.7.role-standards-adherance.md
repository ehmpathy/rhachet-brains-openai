#### rule.require.narrative-flow

**requirement:** flat linear code paragraphs, no nested branches

**check:** output extraction (lines 262-299)
```ts
const output = (() => {
  // if no content, return null (brain deferred to tools)
  if (!content) return null;

  // when tools are plugged, we skip response_format
  if (hasTools && isObjectSchema) {
    try {
      return askInput.schema.output.parse(JSON.parse(content));
    } catch {
      // try to extract JSON from response
      ...
      if (calls) return null;
      return null;
    }
  }

  // standard parse
  try {
    return ...
  } catch {
    if (calls) return null;
    throw new Error('structured output parse failed with no tool calls');
  }
})();
```

**concern:** nested try/catch within if block. however, this is acceptable for parse logic with fallback. the narrative flows: check → try → fallback → throw.

**verdict:** ✓ acceptable — error fallback necessitates try/catch

#### rule.forbid.failhide

**requirement:** never hide errors, always rethrow or handle explicitly

**check:** catch blocks in output extraction
```ts
} catch {
  // extracted JSON didn't match schema
}
```

**concern:** empty catch block at line 278. this is acceptable because:
1. we try multiple parse strategies (full JSON, extracted JSON)
2. if all fail and calls present, we return null (expected behavior)
3. if all fail and no calls, we throw (line 297)

**verdict:** ✓ acceptable — error is not hidden, fallback logic is explicit
