# self-review: has-pruned-backcompat

## backwards compatibility concerns in blueprint

### 1. "[○] retain" markers

the blueprint uses `[○] retain` for 12 code elements:
- config lookup by slug
- BrainAtom constructor
- ask method signature
- start time capture
- system prompt composition
- openai client from context
- zod schema conversion
- prior episode exchanges
- model, input
- message item handle
- metrics calculation
- input/output for message exchanges

**question**: was this backwards compat explicitly requested?

**answer**: yes. the behavior declaration makes this explicit.

the vision before/after table states:
```
| before | after |
| output always present | output nullable when brain defers |
| calls always null | calls contains tool invocations when plugged |
```

this declares the contract: when no tools are plugged, output stays always-present and calls stays null. the retain markers are required to fulfill this.

**why each retain marker holds**:

1. **config lookup by slug** — core mechanism for atom instantiation. without this, atoms cannot be created. not a backwards compat choice, but a core requirement.

2. **BrainAtom constructor** — the constructor signature is part of the public api. tools are passed via `ask()` not constructor.

3. **ask method signature** — TPlugs generic extends the signature, does not replace it. string prompt remains valid.

4. **start time capture** — required for metrics.cost.time calculation. metrics contract is unchanged.

5. **system prompt composition** — tool definitions go into the api call, not the system prompt. composition logic unchanged.

6. **openai client from context** — dependency injection pattern. tools don't change how we obtain the client.

7. **zod schema conversion** — still needed for output parse when brain responds (not tools). the schema applies to final output, not tool invocations.

8. **prior episode exchanges** — continuation works identically. tools add new exchange types (tool call/result) but don't replace message exchanges.

9. **model, input** — api call structure unchanged. tools are additive parameters.

10. **message item handle** — when brain responds with text (no tools), we still extract the message. this path must remain.

11. **metrics calculation** — token counts work the same. openai returns usage regardless of tools.

12. **input/output for message exchanges** — episode serialization for text exchanges unchanged. tool exchanges use same structure with different content.

**verdict**: retain markers are required by vision. each preserves core functionality that tools build upon, not replaces.

### 2. test case: "no tools plugged"

the blueprint includes test case 4: "no tools plugged → extant behavior → output defined, calls null"

**question**: was this backwards compat explicitly requested?

**answer**: yes. blackbox criteria explicitly includes:

```
# usecase.3 = no tools plugged

given(a brain atom without tools plugged)
  when(ask is called)
    then(output is always defined)
      sothat(extant behavior preserved)
    then(calls is always null)
      sothat(type system enforces no null checks)
```

the sothat clause says "extant behavior preserved" — backwards compat was explicitly scoped.

**why this test case holds**:

the test verifies that atoms without tools continue to work as before. this is not defensive "to be safe" — it's explicit in the criteria.

if we removed this test:
- callers with no tools would have no verification
- regression could silently break extant use cases
- the sothat clause would be unfulfilled

the criteria demands we prove extant behavior works. the test fulfills that demand.

**verdict**: test case required by criteria. no change.

### 3. response_format conditional

the blueprint conditionally applies response_format only when no tools are plugged:

```ts
...(!hasTools && isObjectSchema && {
  text: {
    format: { type: 'json_schema', ... },
  },
}),
```

**question**: was this backwards compat explicitly requested?

**answer**: yes. the wish states "w/ and w/o output schema" which implies structured output should continue to work when no tools are plugged. the response_format conditional preserves this.

**why this conditional holds**:

the wish explicitly scopes two scenarios:
1. "w/ output schema" — structured output via response_format (extant)
2. "w/o output schema" — unstructured output (extant)

plus tools:
3. with tools — tool invocation flow (new)

the conditional ensures scenarios 1 and 2 remain unchanged when no tools are plugged. this is not assumed "to be safe" — the wish explicitly listed both cases.

additionally, the research discovered response_format and tools are mutually exclusive in openai's api. the conditional is required for correct api usage, not just backwards compat.

**verdict**: required by wish and api constraint. no change.
