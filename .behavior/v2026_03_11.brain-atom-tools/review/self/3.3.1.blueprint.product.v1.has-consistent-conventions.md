#### why `openaiFunctionTools` holds

extant pattern: no prefix namespace for internal variables (e.g., `jsonSchema`, `fullPrompt`)

the `openai` prefix is warranted because:
- distinguishes from rhachet's `BrainPlugToolDefinition[]` (input)
- clarifies this is the transformed format for the api
- follows the pattern of `OpenAI.Responses.ResponseInputItem` type reference

#### why `isToolExecutionArray` holds

extant pattern: `isObjectSchema` for type/state boolean

this follows the same pattern:
- checks if prompt is the array variant
- `is` prefix for "is this X?" question
- descriptive suffix explains what X is

#### why `functionCallItems` holds

extant pattern: `outputItem` (singular for find), `priorMessages` (plural for array)

the plural form is correct because:
- filter returns array, not single item
- multiple function calls can occur in one response
- `Items` suffix clarifies this is a collection

#### analysis: `hasTools` vs `isX` pattern

extant uses `isObjectSchema` (type check). blueprint uses `hasTools` (property existence).

**semantic difference**:
- `isX` = "is this X?" — checks type or state
- `hasX` = "does this have X?" — checks property existence

**in context**:
- `isObjectSchema` = "is the schema an object type?" — type question
- `hasTools` = "does plugs have tools?" — existence question

**verdict**: both are correct boolean patterns. `hasTools` accurately describes "does askInput.plugs have tools defined?" — this is an existence check, not a type check.

**why it holds**: `isToolsPresent` or `isToolsPlugged` would also work, but `hasTools` is more direct and follows the "has property" pattern common in JavaScript (`object.hasOwnProperty`).

searched for alternatives:
- `isToolsPlugged` — less direct, adds "plugged" verb
- `hasPlugsTools` — matches object path but awkward
- `toolsPlugged` — not a boolean prefix pattern

`hasTools` is the most direct expression of the check.

### term consistency

| blueprint term | rhachet term | openai term | verdict |
|----------------|--------------|-------------|---------|
| `slug` | `slug` | `name` | uses rhachet's term ✓ |
| `exid` | `exid` | `call_id` | uses rhachet's term ✓ |
| `input` | `input` | `arguments` | uses rhachet's term ✓ |
| `output` | `output` | `output` | same ✓ |

**why term consistency holds**:

the blueprint is a rhachet provider implementation. rhachet defines the domain language:
- callers use rhachet terms (`slug`, `exid`, `input`)
- implementation translates to provider terms (`name`, `call_id`, `arguments`)

if we used openai terms in the domain layer:
- callers would need to know openai internals
- switching providers would require api changes
- domain boundary would be violated

the openai map table in the blueprint documents the translation, which keeps the translation explicit and reviewable.
