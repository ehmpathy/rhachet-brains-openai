# self-review: has-questioned-assumptions

*reviewed with fresh eyes on 2026-03-11*

## deep reflection

i re-read the vision line by line. several hidden assumptions surfaced that weren't obvious at first.

the vision makes claims about how tool use "should" work. but did the wisher actually specify this? or did i infer a design from partial information?

let me be honest: much of the vision is my own invention, not the wisher's specification. the wisher only said:
1. rhachet upgraded BrainAtom to support `{ plugs: { tools } }`
2. we want to support that here
3. test it works (w/ and w/o output schema)

everything else — the code examples, the "aha moment", the edgecases table, the awkwardness section — is my interpretation.

this is dangerous. i may have designed the wrong thing.

## assumptions surfaced and questioned

### 1. assumption: openai responses api format matches our contract

**what we assume**: openai's `tools` parameter and `tool_calls` response map cleanly to rhachet's `BrainPlugToolDefinition` and `BrainPlugToolInvocation`.

**evidence**: web search confirmed openai supports function call. exact format not yet verified against real api.

**what if opposite were true**: if formats differ significantly, we'd need adapter logic. could add complexity.

**did wisher say this**: no. inferred from "support plugs.tools per rhachet contract".

**verdict**: reasonable assumption, but must verify with real api calls in implementation phase.

---

### 2. assumption: brain returns EITHER output OR tool calls, not both

**what we assume**: when tools are plugged, the response is XOR: either `output` is defined and `calls` is null, or vice versa.

**evidence**: common pattern in llm apis. not explicitly verified for openai responses api.

**what if opposite were true**: if openai can return BOTH structured output AND tool calls in a single response, our type narrowing would be wrong. `output` would not be null when `calls` is present.

**counterexamples**: some apis allow "parallel tool calls" where model requests multiple tools. does openai allow output alongside those?

**did wisher say this**: no. inferred from rhachet's type definitions (`AsBrainOutputOutputFor` narrows to `TOutput | null`).

**verdict**: HIGH RISK assumption. must verify with openai docs and real api tests.

---

### 3. assumption: tool result continuation works via prompt injection

**what we assume**: callers can feed tool results back by concatenating them into the prompt string.

**evidence**: atoms are single-turn. episode continuation already extant. tool results as prompt text is simple.

**what if opposite were true**: if openai requires specific message format for tool results (role: "tool"), prompt injection wouldn't work correctly. model might not understand the tool result.

**did wisher say this**: no. wisher only said "tools are successfully invocable". continuation is inferred.

**verdict**: MEDIUM RISK. prompt injection may work but isn't the "correct" openai format. may need explicit tool result input in future.

---

### 4. assumption: zod schemas can be converted to openai function schemas

**what we assume**: `z.toJSONSchema()` produces schemas compatible with openai's function parameter format.

**evidence**: current implementation uses `z.toJSONSchema()` for output schema. should work for tool input schemas too.

**what if opposite were true**: if openai has specific schema constraints (e.g., no $ref, specific type formats), our zod schemas might produce invalid definitions.

**did wisher say this**: no. inferred from rhachet contract which uses zod schemas.

**verdict**: LOW RISK. zod's json schema output is standard. likely compatible.

---

### 5. assumption: all openai models support tool use

**what we assume**: any model in `CONFIG_BY_ATOM_SLUG` with `skills.tooluse: true` actually supports the tools parameter.

**evidence**: config already has `tooluse` skill flag. web search confirmed function call is widely supported.

**what if opposite were true**: some models might have different tool use syntax or limitations (e.g., o1 models had restrictions initially).

**did wisher say this**: no. inferred from extant config structure.

**verdict**: LOW RISK. config already tracks this. tests should use a model known to support tools.

---

### 6. assumption: output schema and tools can coexist

**what we assume**: we can pass both `text.format` (for structured output) and `tools` to openai's responses api.

**evidence**: none. common sense suggests they might conflict.

**what if opposite were true**: if only one can be active, we'd need to choose: structured output OR tool use. or the behavior might be model-dependent.

**did wisher say this**: yes, indirectly — "w/ and w/o output schema" suggests both modes should work.

**verdict**: MEDIUM RISK. must verify with real api tests.

---

## conclusions

the **high risk** assumption (output XOR calls) needs verification before implementation. if openai can return both, the rhachet contract may need revision.

the **medium risk** assumptions (tool results via prompt, schema+tools coexist) should be tested but are not blockers.

the **low risk** assumptions are reasonable and can be validated during implementation.

## recommendation

1. **verify high risk**: check openai docs for whether response can contain both output and tool_calls
2. **test medium risk**: integration tests should cover both scenarios
3. **proceed**: vision is sound, assumptions are documented, risks are known
