## extant conventions identified

### file name patterns

| pattern | purpose | examples |
|---------|---------|----------|
| `gen*.ts` | generators that produce new instances | `genBrainAtom.ts`, `genBrainRepl.ts` |
| `cast*.ts` | type/format conversions | new files |
| `as*.ts` | value conversions (wraps, transforms) | `asJsonSchema.ts` |
| `*.config.ts` | configuration objects | `BrainAtom.config.ts` |
| `*.integration.test.ts` | integration tests | `genBrainAtom.integration.test.ts` |
| `*.test.ts` | unit tests | `castIntoOpenaiFunctionTool.test.ts` |

### cast directional convention

| pattern | direction | intent |
|---------|-----------|--------|
| `castInto*` | out of domain | rhachet domain → external format |
| `castFrom*` | into domain | external format → rhachet domain |

this directional convention makes the cast boundary explicit at the function name level.

### comment convention

```ts
/**
 * .what = intent summary
 * .why = purpose explanation
 * .note = optional caveat
 */
```

### variable name convention

follows `[noun][state]` order per briefs:
- `functionCallItems` (noun)
- `exchangeInput`, `exchangeOutput` (noun + qualifier)
- `customerFound` instead of `foundCustomer`
