### 2. do we need both integration AND acceptance tests?

**quote from wish**: "cover with integration and acceptance tests that tools are successfully invocable"

**question**: what's the distinction? for a brain package:

### 3. "w/ and w/o output schema" — what does this mean?

**quote from wish**: "w/ and w/o output schema"

**question**: does this mean:
- a) test tool use WITH structured output schema AND without (raw string)?
- b) test scenarios where tools are defined vs not defined?
- c) other?

**answer**: likely (a) — the current implementation supports both `z.object({})` (json schema) and `z.string()` (raw output). with tools, we should test both modes.

**risk if wrong scope**: we'd miss a test case or over-test.

### 4. is tool result continuation in scope?

**observation**: the vision assumes callers feed tool results back via prompt strings:

```ts
prompt: `tool result: ${JSON.stringify(toolResult)}`
```

but openai's responses api has a specific message format for tool results (role: "tool", tool_call_id: "...").

**question**: should the atom support explicit `toolResults` input? or is prompt injection sufficient?

**answer**: for v1, prompt injection is simpler and matches the atom's single-turn design. explicit tool result injection would be a follow-up enhancement if needed.

**tradeoff acknowledged**: this is awkward but keeps the contract simple.

---

### 5. is the vision too detailed?

**observation**: the vision includes implementation details (code examples, type definitions, exact flow).

**question**: should a vision document be more abstract?

**answer**: for a technical library, concrete examples help validate the contract makes sense. the vision serves as a "design doc" not just a "dream". the detail level is appropriate.

---

### 6. what if tools and output schema conflict?

**question**: can openai return both a structured json response AND tool calls? or is it one or the other?

**assumption in vision**: brain either returns output OR requests tools, not both.

**need to verify**: does openai's api allow both `response_format` (structured output) and `tools` simultaneously? if so, can a single response contain both structured output AND tool calls?

**impact**: this affects the type contract — is `output` truly nullable when tools are plugged, or could we get both?
