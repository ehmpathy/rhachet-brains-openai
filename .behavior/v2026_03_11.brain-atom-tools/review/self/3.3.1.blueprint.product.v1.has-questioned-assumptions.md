# self-review: has-questioned-assumptions

## assumptions surfaced and questioned

### 1. response_format and tools are mutually exclusive

**assumption**: when tools are plugged, we must skip response_format.

**evidence**:
- openai docs: "function call is for tool use; structured outputs via response_format for user responses"
- research document 3.1.1.research.external.product.claims confirmed: "structured outputs vs tools serve different purposes"
- openai community confirms this pattern

**what if opposite?**: if we could use both, we'd get structured final output AND tool calls. but api design separates these concerns.

**verdict**: validated assumption. proceed with mutual exclusivity.

---

### 2. tool.slug maps to function.name

**assumption**: we use `tool.slug` as `function.name` in openai format.

**question**: should we use `tool.name` instead?

**evidence**:
- openai expects `name` field for function identification
- rhachet's `slug` is the unique identifier (e.g., "stripe.customer.lookup")
- rhachet's `name` is human-readable display name
- invocations return `name` which we need to map back to slug

**what if opposite?**: if we use tool.name, we'd need to maintain a separate lookup. slug is already unique and used for dispatch.

**verdict**: validated assumption. slug works as function.name since it's unique and used for lookup.

---

### 3. BrainPlugToolExecution.output needs JSON.stringify

**assumption**: we stringify the output for function_call_output.

**evidence**:
- openai expects `output` as string
- BrainPlugToolExecution.output is typed TOutput (any shape)
- must serialize for transmission

**simpler approach?**: could check if already string, but JSON.stringify handles all cases uniformly.

**verdict**: validated assumption. stringify is correct and uniform.

---

### 4. function_call.arguments is JSON string

**assumption**: we JSON.parse the arguments field.

**evidence**:
- openai docs show arguments as string that contains JSON
- research document shows: `"arguments": { "location": "Paris" }` in examples

**verified**: checked SDK types (openai/resources/responses/responses.d.ts):
```ts
ResponseFunctionToolCall {
  /**
   * A JSON string of the arguments to pass to the function.
   */
  arguments: string;  // confirmed: is JSON string
  type: 'function_call';
}
```

**verdict**: validated assumption. SDK returns arguments as JSON string, requires JSON.parse.

---

### 5. episode exchange stores tool calls

**assumption**: tool calls become exchanges in episode for continuation.

**question**: what should the exchange input/output be for tool calls?

**verified**: checked BrainExchange type (rhachet/domain.objects/BrainExchange.d.ts):
```ts
interface BrainExchange {
  input: string;   // user's request
  output: string;  // brain's response
  exid: string | null;
}
```

**approach**: for tool calls, serialize the invocations as JSON for the output field:
```ts
exchange: {
  input: askInput.prompt,  // original prompt or serialized tool results
  output: JSON.stringify(calls.tools),  // serialized tool invocations
  exid: response.id,
}
```

this maintains compatibility with the string-based exchange contract and preserves tool call data for continuation.

**verdict**: validated assumption. serialize tool invocations as JSON string for output field.

---

### 6. metrics calculation unchanged

**assumption**: token count works the same with tools.

**evidence**:
- api still returns `usage.input_tokens` and `usage.output_tokens`
- tool definitions add to input tokens
- tool invocations add to output tokens

**verdict**: validated assumption. api handles token account.
